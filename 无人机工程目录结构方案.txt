CH32H417 无人机工程目录结构方案

目标：
给你们做一个适合比赛开发、后期可维护、又不会一开始过度复杂化的工程结构。

设计原则：
1. 先服务于“能飞起来”
2. 驱动、算法、应用分层清楚
3. 便于后面逐步加悬停、导航、投放、质量辨识
4. 兼容你们当前“部分外设/信息来自另一颗芯片”的现状

==================================================
一、推荐的工程总目录
==================================================

project_root/
├─ app/
├─ algo/
├─ drivers/
├─ bsp/
├─ middleware/
├─ common/
├─ config/
├─ test/
├─ tools/
├─ docs/
├─ third_party/
└─ EVT_ref/

下面逐项解释。

==================================================
二、各目录职责
==================================================

1. bsp/
职责：
- 板级支持包
- 直接面向 CH32H417 外设寄存器/标准库
- 做最底层硬件初始化

建议子目录：

bsp/
├─ clock/
├─ gpio/
├─ pwm/
├─ uart/
├─ spi/
├─ i2c/
├─ adc/
├─ dma/
├─ exti/
├─ timer/
└─ watchdog/

建议文件命名：
- bsp_pwm.c / bsp_pwm.h
- bsp_uart.c / bsp_uart.h
- bsp_spi.c / bsp_spi.h
- bsp_i2c.c / bsp_i2c.h

这一层只做两件事：
- 初始化外设
- 提供简单可靠的硬件访问接口

不要在这一层里写：
- 飞行模式
- PID
- 导航逻辑
- 投放逻辑

2. drivers/
职责：
- 面向具体设备芯片/模块
- 调用 bsp 层完成读写

建议子目录：

drivers/
├─ imu/
├─ nrf24/
├─ optflow/
├─ tof/
├─ gps/
├─ magnet/
├─ battery/
├─ vision_link/
├─ baro_link/
└─ payload/

各目录建议职责：
- imu/：六轴 IMU 数据接收、解析、标定
- nrf24/：遥控数据收发、丢包检测
- optflow/：光流模块数据解析
- tof/：TOF 距离读取、滤波、有效性判断
- gps/：GPS 数据解析
- magnet/：电磁铁控制驱动
- battery/：电压/电流采样换算
- vision_link/：与视觉芯片的通信协议
- baro_link/：接收另一颗芯片转发的气压计/高度信息
- payload/：投放器状态、载荷状态

说明：
- 你们的气压计不直接归 H417 管，所以我建议不要硬塞进某个本地 I2C 传感器驱动。
- 单独做一个 baro_link/，专门表示“高度信息来自外部处理器”。
- 这样架构更清楚。

3. algo/
职责：
- 放纯算法，不直接操作硬件
- 输入是传感器数据，输出是估计量或控制量

建议子目录：

algo/
├─ filter/
├─ attitude/
├─ altitude/
├─ position/
├─ control/
├─ estimator/
└─ math/

建议内容：
- filter/：低通、滑动平均、一阶滤波、互补滤波
- attitude/：姿态解算（Mahony / Madgwick / complementary）
- altitude/：TOF + 气压计 + IMU 的高度估计
- position/：光流 + 姿态补偿的位置/速度估计
- control/：PID、限幅、抗积分饱和
- estimator/：质量估计、推力模型、扰动估计
- math/：向量、矩阵、坐标变换

注意：
- 质量辨识建议放在 estimator/ 下，不要跟基础 PID 混一起。

4. app/
职责：
- 这里是系统行为层
- 决定“什么时候做什么”

建议子目录：

app/
├─ flight/
├─ mission/
├─ safety/
├─ calibration/
├─ payload/
└─ telemetry/

建议职责：
- flight/：飞行主循环、模式切换、状态机
- mission/：导航任务、投放任务
- safety/：失联保护、低电保护、起飞前检查
- calibration/：IMU 校准、遥控校准、传感器零偏管理
- payload/：电磁铁释放逻辑、投放判据
- telemetry/：调试输出、状态回传

5. middleware/
职责：
- 放协议层、通用缓冲区、消息编解码
- 介于 driver 和 app 之间

建议子目录：

middleware/
├─ protocol/
├─ message_bus/
└─ ringbuffer/

适合放：
- 视觉板通信协议
- 外部芯片高度数据协议
- 遥测帧格式
- CRC/校验

6. common/
职责：
- 全工程都可能复用的小工具

建议内容：
- common_def.h
- error_code.h
- macros.h
- util_time.c
- util_math.c

7. config/
职责：
- 参数、编译配置、机型配置

建议文件：
- board_config.h
- flight_config.h
- pid_config.h
- sensor_config.h
- mission_config.h

建议把这些东西放这里：
- 电机引脚映射
- 混控方向
- 遥控通道映射
- 传感器安装方向
- PID 默认参数
- 高度/速度阈值
- 投放判据阈值

8. test/
职责：
- 单模块联调
- 地面测试代码

建议子目录：

test/
├─ test_pwm/
├─ test_imu/
├─ test_nrf/
├─ test_optflow/
├─ test_tof/
├─ test_baro_link/
├─ test_motor_mix/
└─ test_drop/

这里不是 PC 单元测试的意思，而是嵌入式分模块验证工程/代码集合。

9. tools/
职责：
- 调参脚本
- 串口日志抓取脚本
- 数据解析脚本
- 绘图脚本

建议内容：
- Python 串口画图脚本
- 日志转 CSV 工具
- PID 调试小脚本

10. docs/
职责：
- 放你们真正和项目开发强相关的文档

建议内容：
- 传感器协议说明
- 引脚分配表
- 飞控模式说明
- 调参记录
- 状态机流程图

不要把大量无关资料堆进来。

11. third_party/
职责：
- 官方库、外部库

建议内容：
- CH32H417 标准外设库
- 你们引用的第三方滤波/数学库

12. EVT_ref/
职责：
- 只放少量你们实际参考过的 EVT 例程
- 不要把整个 EVT 再拷进来

建议只保留：
- TIM/PWM_Output
- USART/USART_DMA
- USART/USART_Interrupt
- SPI/2Lines_FullDuplex
- GPIO/GPIO_Toggle
- EXTI/EXTI0
- DMA/DoubleBuffer_DMA（如以后确实需要）

==================================================
四、主循环建议怎么分
==================================================

建议采用“分频调度”思路，不要所有东西塞一个 while(1)。

可以参考：

- 1kHz：IMU 接收 / 内环准备
- 500Hz：角速度控制
- 200Hz：姿态控制
- 100Hz：TOF/光流/遥控更新
- 50Hz：高度控制 / 位置控制
- 20Hz：GPS / 视觉任务 / 状态上报
- 10Hz：日志 / 状态灯 / 慢速监控

即使前期不做 RTOS，也可以先做一个简单的软件调度器。

==================================================
五、针对“另一颗芯片提供视觉和气压计”的结构建议
==================================================

这个是你们项目里很关键的特点。

我建议明确拆成两个输入链路：

1. vision_link
输入内容建议包含：
- target_found
- target_type
- target_dx
- target_dy
- confidence
- allow_drop
- frame_seq
- timestamp

2. baro_link
输入内容建议包含：
- baro_height
- vertical_speed（如果对方可提供）
- valid_flag
- fault_flag
- seq
- timestamp

重要建议：
- 一定不要只发一个裸数值。
- 你们后面做融合、查 bug、分析延迟时，会非常依赖 seq 和 timestamp。

==================================================
八、你们这个项目最容易犯的目录设计错误
==================================================

1. 把所有代码都写进 main.c
结果：
- 后期完全不可维护

2. 驱动和算法混在一起
结果：
- 以后换传感器特别痛苦

3. 视觉通信、气压计转发、遥控协议都各写各的，数据结构不统一
结果：
- 状态判断混乱
- 联调极难

4. 校准参数和 PID 参数到处散落
结果：
- 不知道最终生效的是哪一份

5. 一开始就想把双核、导航、识别、投放、质量辨识一起塞进去
结果：
- 系统太复杂，谁也调不稳

==================================================
九、最适合你们当前阶段的工程落地建议
==================================================

如果你们现在马上要开始搭主工程，我建议：

第一阶段工程只保留这些大模块：
- bsp/
- drivers/imu
- drivers/nrf24
- algo/attitude
- algo/control
- app/flight
- app/safety
- config/
- common/

等你们“能稳定飞”后，再加：
- drivers/optflow
- drivers/tof
- drivers/baro_link
- algo/altitude
- algo/position

再后面再加：
- drivers/vision_link
- app/payload
- app/mission
- algo/estimator

也就是说：
- 先搭最小飞控骨架
- 再逐层扩展
- 不要一开始把最终架构全部堆满

==================================================
十、最终一句建议
==================================================

如果你愿意把这个项目认真做成“能比赛”的工程，而不是“能跑几个例程的集合”，那最关键的不是代码写得多快，而是从一开始就把目录职责分清楚。

对你们这个项目来说，最值得强调的是：
- 飞控主链路要简洁
- 高度/视觉这种跨芯片信息一定要协议化
- 先飞稳，再做高级功能

这套目录结构就是按这个思路给你们定的。
