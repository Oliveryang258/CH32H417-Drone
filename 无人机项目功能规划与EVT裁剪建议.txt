CH32H417 无人机项目功能规划与 EVT 裁剪建议

一、先说结论

按你现在描述的目标，你们现有外设“能支撑项目起步”，但还不算“稳稳够用”。

你们现在已有：
1. 匿名光流
2. 串口六轴 IMU（加速度计 + 陀螺仪）
3. TOF 测距
4. 气压计（不归 CH32H417 直接管理）
5. GPS（室外备用）
6. NRF 遥控通信
7. 另一个芯片负责摄像头识别
8. 电磁铁投放机构

这套配置已经可以支撑：
- 四轴基础飞行
- 姿态稳定
- 高度估计与高度保持
- 简单起飞/降落
- 室内低速相对移动
- 识别后触发投放

但是如果你们想把“起飞悬停、定向飞多少米、识别后精确投放、投放后重心突变仍稳定”真正做稳，当前最明显的短板已经不再是高度，而是下面两个问题：

A. 航向长期稳定能力不足
- 你们现在说的是“六轴 IMU”，一般表示只有陀螺仪 + 加速度计，没有磁力计。
- 没有磁力计时，yaw（航向角）会漂移。
- 这会直接影响“朝固定方向飞多少米”这个目标，因为你以为机头朝前，实际可能慢慢偏了。

B. 多源高度信息与跨芯片协同复杂度
- 你们虽然已经有 TOF 和气压计，这很好，但气压计不归 CH32H417 直接管理。
- 这意味着高度相关信息可能要通过另一颗芯片转发给飞控主控，系统复杂度会上升。
- 如果跨芯片通信时延、刷新率、时间戳或丢包处理做不好，高度环仍然会受影响。

所以我的判断是：
- 如果比赛目标是“先飞起来 + 室内相对悬停 + 简单投放”，当前外设已经足够做第一版。
- 如果比赛目标是“比较稳的悬停 + 定向飞行 + 投放后快速恢复稳定”，当前最建议补的外设不再是高度传感器，而是：
  1）磁力计（强烈建议）
  2）电池电压/电流采样（强烈建议）


二、我对你们现有外设是否足够的详细判断

1. IMU（六轴）
作用：
- 角速度测量
- 姿态解算
- 飞控内环稳定控制

结论：
- 必须有，而且这是飞控核心。
- 只靠六轴可以做基础姿态稳定。
- 但只靠六轴不适合长期可靠航向保持。

建议：
- 先用它完成姿态环、角速度环。
- 后期最好补磁力计，或者让视觉板提供航向修正信息。

2. 光流 + TOF
作用：
- 室内/近地相对位移估计
- 水平速度估计
- 辅助悬停
- 为光流速度换算提供高度尺度

结论：
- 对室内悬停非常有帮助。
- 你们既然已经有 TOF，那么高度保持这条链路比“只有光流”时要可靠得多。
- 这意味着你们更适合先把室内悬停和相对导航作为主攻方向。

关键判断：
- 现在关键不再是“缺不缺高度传感器”，而是 TOF 的刷新率、量程、地面材质适应性，以及与飞控主循环的同步。
- 如果 TOF 只在低高度范围内稳定，那高于量程后要设计降级策略。

3. 气压计（由其他芯片管理）
作用：
- 中高高度趋势估计
- 低频高度漂移修正
- 与 TOF 互补

结论：
- 这是有价值的，但复杂点在于它不归 CH32H417 直接管理。
- 如果另一颗芯片只是偶尔转发高度值，而没有时间戳、更新标记、异常标志，那么融合效果会变差。

建议：
- 尽量让另一颗芯片向 H417 输出结构化数据，而不是只发一个裸高度值。
- 最好一起发：高度值、时间戳/序号、数据有效标志、异常状态。
- 在低空时优先信任 TOF，在较高处或 TOF 失效时再提高气压计权重。

4. GPS
作用：
- 室外粗略定位
- 室外航点飞行辅助

结论：
- 可以作为室外模式补充。
- 但普通 GPS 通常精度不够，不适合高精度定点悬停和精准投放。
- 如果比赛主要在室内，GPS 不是第一优先级。
- 如果比赛在室外，GPS 可以做“粗导航”，但不要过分依赖它做精确投放。

建议：
- 先不把 GPS 作为首要攻坚点。
- 先把 IMU + 光流 + 高度保持做稳。
- GPS 放到第二阶段整合。

4. NRF 遥控通信
作用：
- 遥控指令接收
- 模式切换
- 急停/解锁/复位等控制

结论：
- 足够。
- 但要重点做好丢包保护和失联保护。

必须实现：
- 遥控失联检测
- 失联后的降落/锁桨/降功率策略
- 通信数据合法性校验
- 解锁保护

5. 摄像头在另一个芯片上处理
作用：
- 图像识别
- 目标检测
- 将识别结果发送给 H417 飞控

结论：
- 这是合理的系统划分。
- 对 H417 来说，不一定需要自己跑图像，只需要接收“识别结果”。

但这里有一个关键点：
- 另一个芯片不能只告诉你“看到了目标”。
- 最好还能输出：
  1）目标是否识别成功
  2）目标在图像中的偏移量（左右偏多少、前后偏多少）
  3）识别置信度
  4）是否满足投放条件

否则飞控只能“看见就扔”，投放精度会差很多。

6. 电磁铁
作用：
- 吸附物体
- 投放时断电释放

结论：
- 原理上没问题。
- 但不能直接由 MCU 引脚硬驱动，必须有功率驱动电路。

必须注意：
- MOSFET 驱动
- 续流二极管
- 与飞控供电隔离/去耦
- 防止电磁铁通断导致供电波动、干扰 IMU、复位 MCU

建议增加：
- 电磁铁状态指示
- 如果条件允许，可测一下吸合电流，便于判断是否真正工作


三、我建议补充的外设

如果你问“为了比赛更稳，最应该补什么”，优先级如下：

第一优先级：磁力计
- 原因：解决 yaw 漂移，改善“朝固定方向飞”的能力。
- 没有磁力计也能飞，但定向和稳定导航会难很多。

第二优先级：电池电压/电流采样
- 这对飞行安全很重要。
- 也对你们后面做质量辨识、推力补偿、电池压降补偿很有帮助。

第三优先级：黑匣子/日志接口
- 可以是串口日志，也可以是简化数据记录。
- 用于调 PID、查炸机原因、看估计量是否发散。

第四优先级：跨芯片高度/视觉数据协议规范化
- 因为你们的气压计和摄像头都不完全在 H417 本芯片侧。
- 如果跨芯片通信协议不规范，再好的传感器也发挥不稳定。


四、为了完成你们的目标，需要实现哪些功能

建议你们按阶段做，不要一开始就同时做“飞行 + 导航 + 识别 + 投放 + 质量辨识”。

====================
阶段 0：最小可飞系统
====================
目标：先安全转桨、受控输出、遥控可用、姿态数据可信。

必须完成：
1. 时钟与系统初始化
2. 四路电机 PWM 输出
3. 遥控 NRF 通信
4. IMU 串口驱动
5. IMU 数据解析
6. 基本校准（零偏、静态校准）
7. 电机解锁/上锁逻辑
8. 基本保护逻辑
9. 串口调试输出

对应功能模块：
- board 层：时钟、GPIO、PWM、串口、SPI、DMA、中断
- driver 层：NRF、IMU、光流、GPS、电磁铁、ADC
- app 层：解锁、模式切换、故障保护

====================
阶段 1：基础飞行控制
====================
目标：手动模式下稳定飞起来。

必须完成：
1. 姿态解算
- 可先做 Mahony / Madgwick / 互补滤波
- 没必要一开始就上很重的 EKF

2. 电机混控
- X 型四旋翼混控矩阵
- 油门、横滚、俯仰、偏航到四电机分配

3. 控制器
- 角速度环（内环）
- 姿态角环（外环）
- 限幅与积分保护

4. 遥控映射
- 油门、roll、pitch、yaw、模式开关

5. 安全逻辑
- 上电自检
- 遥控失联处理
- 电机保护
- 起飞前传感器就绪检测

建议目标：
- 先达到“手动模式能稳定飞、不会明显自激、落地不炸机”。

====================
阶段 2：悬停与相对导航
====================
目标：让飞机能悬停，并按相对方向移动固定距离。

必须完成：
1. 光流数据读取与标定
2. 高度估计
- 由 TOF/测距/气压计提供主信息
- IMU 做辅助

3. 水平速度/位移估计
- 光流 + 姿态补偿
- 必要时融合加速度信息

4. 悬停控制
- 高度环
- 水平速度环/位置环

5. 航向保持
- 没磁力计时可以先做短时间 yaw hold
- 但长时间效果一般

6. 相对导航
- 向前飞 N 米
- 向左/向右飞 N 米
- 定时、定速、定距离策略

建议目标：
- 室内能做低速定向移动
- 能悬停 3~5 秒以上且不明显飘走

====================
阶段 3：识别与投放
====================
目标：识别目标后在合适位置投放。

必须完成：
1. 与视觉板通信协议
- 目标类型
- 目标偏移量
- 识别置信度
- 投放允许标志

2. 投放任务状态机
- 搜索目标
- 靠近目标
- 对准目标
- 释放电磁铁
- 脱离目标区

3. 电磁铁驱动控制
- 吸合
- 断电释放
- 保护延时

4. 投放判据
- 不是“识别到就扔”
- 应该是“位置误差进入阈值 + 高度符合 + 姿态稳定 + 速度足够小”再投

建议目标：
- 先做“识别成功后手动确认投放”
- 再做“全自动投放”

====================
阶段 4：质量突变与鲁棒性增强
====================
目标：投放后机体质量突变时仍能保持稳定。

必须完成：
1. 电池电压补偿
2. 油门-推力近似模型
3. 竖直方向加速度/速度估计优化
4. 质量变化检测逻辑
5. 参数在线调整或前馈补偿

你们提到的最小二乘参数辨识法，可以放在这一阶段。

我的建议是：
- 不要把“在线质量辨识”当作第一阶段任务。
- 更合理的顺序是：
  先飞稳 -> 再悬停 -> 再投放 -> 最后做质量突变补偿。

因为如果基础姿态环和高度环都不稳，质量辨识的结果没有意义。


五、你们的软件架构建议

建议至少拆成下面几层：

1. BSP / HAL 层
- PWM
- GPIO
- USART
- SPI
- I2C（若以后加磁力计/气压计）
- ADC
- EXTI
- DMA
- 定时器

2. Driver 层
- drv_imu
- drv_nrf
- drv_optflow
- drv_gps
- drv_magnet
- drv_battery
- drv_vision_link

3. Algorithm 层
- attitude_estimator
- pid
- altitude_estimator
- position_estimator
- mass_estimator
- filter

4. Control / App 层
- flight_mode
- mixer
- failsafe
- mission_manager
- drop_manager
- calibration_manager

5. Debug / Telemetry 层
- 串口输出
- 参数打印
- 日志
- 上位机协议（如果你们后面要做）


六、我建议你们的开发顺序

非常建议按下面顺序推进：

第 1 步：只做单板基础驱动
- PWM 四路正常
- NRF 正常
- IMU 正常
- 串口调试正常

第 2 步：只做姿态稳定
- 不做导航
- 不做投放
- 不做质量辨识
- 目标只是“能稳定受控飞”

第 3 步：做高度保持
- 如果没有高度传感器，这一步会很难做好

第 4 步：做光流悬停
- 先短时间悬停
- 再做相对平移

第 5 步：做视觉通信 + 手动投放
- 先识别后人工确认释放

第 6 步：做全自动投放

第 7 步：做质量辨识与参数补偿

这个顺序的核心思想是：
- 先把飞行器变成“可控平台”
- 再逐个加功能
- 不要一次把所有目标捆在一起


七、如果用 CH32H417，双核到底怎么建议

结合你们目前情况，我的建议是：

第一阶段：尽量先把飞控主链路放在 V3F 上跑通
原因：
- 你们现在还在摸底阶段
- 双核会引入同步、调试、启动顺序、资源分配复杂度
- 比赛项目最怕系统复杂度过高而失控

第二阶段：如果后面算法或通信变重，再考虑把任务拆给 V5F
例如：
- V3F：飞控主循环、PWM、姿态控制、遥控安全
- V5F：视觉结果处理、日志、复杂估计、次要通信

简单说：
- 比赛里先求“稳”，再求“帅”。
- 不建议你们一开始就靠双核结构堆复杂度。
